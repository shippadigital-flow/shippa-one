# Architecture

This document explains the runtime shape, major subsystems, and file locations for the Shippa One application. It is intended to help architects and engineers quickly understand how the app is assembled and where to make changes safely.

High-level runtime flow

- Entry (edge / server): The application exposes a `fetch` entry used for SSR and edge deployments. The entry point is configured via TanStack Start and Nitro (via the Lovable Vite preset).
  - src/server.ts — top-level SSR wrapper. It loads the framework-provided server entry and normalizes catastrophic SSR errors to render a friendly error page.
  - src/start.ts — creates the TanStack Start instance and attaches request middleware (error handling, etc.).

- Routing & Rendering
  - Routing is handled by @tanstack/react-router. A generated route tree (src/routeTree.gen.ts) is used to wire routes into the router.
  - src/router.tsx constructs the router with a QueryClient instance and provides router-level options (scroll restoration, preload settings).
  - Route components render using React and can run on both server and client — TanStack Start provides SSR glue.

- Data Layer
  - Data fetching & caching uses @tanstack/react-query (QueryClient). QueryClient instances are created per router in src/router.tsx; cache behavior should be considered when implementing shared caches across multiple entry points.
  - Services that call external APIs live in src/services. Keep API clients and typed contracts in this directory to maintain separation from UI logic.

- UI Layer
  - A local design system lives in src/components/ui. These components are mostly thin wrappers around Radix primitives, plus Tailwind-based styling.
  - Large components (sidebar, chart, calendar) are present in the UI folder — prefer splitting complex components into smaller subcomponents for maintainability.

- Utilities / error handling
  - src/lib contains server-side helpers, error-page rendering and error-capture utilities used by server.ts and start.ts.

Build system

- Vite is used with a Lovable preset (@lovable.dev/vite-tanstack-config). The preset configures:
  - TanStack devtools and runtime plugins for React Start
  - Nitro target for server builds (edge/Cloudflare default)
  - VITE_* env injection and @ path aliasing

- Entry mapping: vite.config.ts redirects the TanStack Start server entry to `src/server.ts`.

Generated artifacts

- src/routeTree.gen.ts — generated route tree for TanStack Router. This file is generated by the route tooling (router generator) and is committed to the repo for stability, but it is treated as a generated artifact. Do not hand-edit this file.

Key files & references

- src/server.ts — SSR entry wrapper and catastrophic SSR error normalization
- src/start.ts — TanStack Start instance and request middleware
- src/router.tsx — router creation and QueryClient injection
- src/routeTree.gen.ts — generated route definitions
- src/components/ui/ — UI primitives and components
- src/services/ — API clients and business logic
- src/lib/ — helper utilities (error page, error capture)

Operational considerations

- Deployment: the build targets are set up for edge/Nitro. Ensure you understand the target environment (Cloudflare Workers, other edge runtimes) when configuring environment variables and secrets.
- Logging & monitoring: server.ts and error-capture logic print errors to console and render a friendly HTML error page. Integrate with an external observability tool (Sentry, Datadog) for production error aggregation.
- Code generation: regeneration of src/routeTree.gen.ts must be done with the same generator version used to create it; document the generation command in package scripts or local dev docs so contributors know how to update routes.

Recommendations & best practices

- Keep generated files isolated: consider moving generated artifacts to a `generated/` folder or adding a build-time generation step so local hand edits are not accidentally introduced.
- Use feature folders at scale: when features grow, co-locate `pages`, `components`, `hooks`, and `services` for that feature to reduce navigation overhead.
- Enforce unused-checks during CI: enable `noUnusedLocals` and `noUnusedParameters` in CI runs to catch dead code.
- Route-level code-splitting: ensure heavy components are lazy-loaded at the route level to keep initial page bundles small.
